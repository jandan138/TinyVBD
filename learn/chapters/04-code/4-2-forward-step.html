forwardStep()：惯性与自适应初始化

每个 substep 的第一件事就是 forwardStep()。它做两件事：算出惯性预测位置 $\mathbf y$（1.4 拔河里那个「锚」），再为接下来的迭代挑一个聪明的初始猜测 $\mathbf x^{(0)}$。第二件事正是论文里的adaptive initialization——VBD 才有的自由。

重力、pin、惯性 y

函数开头三行就把惯性这一项备齐了：

main.cpp · forwardStep()
strand.mVelocity.colwise() += params.dt * params.gravity;
strand.mVelocity.col(0).setZero();

strand.inertia = strand.mVertPos + strand.mVelocity * params.dt;

strand.mVertPrevPos = strand.mVertPos;
// initial step
// strand.mVertPos = strand.inertia;

逐行读：

  • mVelocity.colwise() += dt * gravity：给每一列（每个顶点）的速度加上重力带来的那一截 $h\,\mathbf g$。这是把外力先积进速度。
  • mVelocity.col(0).setZero()：把第 0 个顶点的速度清零——这就是把 strand 的根 pin 住。整本书里 0 号顶点都是固定挂点（吊着的那一端），它永远不动。4.3 的 solve() 循环从 iV=1 开始、4.5 的 updateVelocity() 也会再清一次，都是为这件事。
  • inertia = mVertPos + mVelocity * dt：这就是惯性预测位置 $\mathbf y$。展开看就是 $\mathbf y = \mathbf x^t + h(\mathbf v^t + h\,\mathbf g) = \mathbf x^t + h\,\mathbf v^t + h^2\,\mathbf g$——和 1.2 给出的 $\mathbf y = \mathbf x^t + h\mathbf v^t + h^2 M^{-1}\mathbf f_{ext}$ 严丝合缝（这里外力只有重力、按加速度给，所以是 $h^2\mathbf g$）。
  • mVertPrevPos = mVertPos：把这一步开始时的位置 $\mathbf x^t$ 存一份，留给末尾反算速度。

注意那行被注释掉的 // strand.mVertPos = strand.inertia;：最朴素的做法就是「直接拿惯性位置当初始猜测」。TinyVBD 没这么干，而是往下走自适应初始化。但它留着这行，等于在告诉你：$\mathbf y$ 是 baseline，下面要做得更好。

inertia 这个字段后面一直在用

strand.inertia（也就是 $\mathbf y$）算完后不会再变，会被 4.3 的 solve() 反复读取：每个顶点的惯性力 f = m*(inertia - x)/dt² 就是拿当前 $\mathbf x$ 和这个固定的 $\mathbf y$ 比。$\mathbf y$ 是这一步里惯性碗的碗底，整步不动；动的是 $\mathbf x$，一点点被推向 $\arg\min G$。

自适应初始化：预判每个顶点该不该「自由下落」

朴素初始化 $\mathbf x^{(0)}=\mathbf y$ 有个隐患：$\mathbf y$ 假设每个顶点都吃满一个 $h^2\mathbf g$ 的重力位移。对正在自由下落的顶点没问题；但对一个被弹簧或接触托住的顶点（比如叠在别人上面、或者被硬弹簧拽着），它本来这一步几乎不该动，你却先把它往下挪了 $h^2 g$——于是初始就过度穿插 / 过度拉伸，迭代还得费劲把它拉回来。论文 Eq.17 的自适应初始化就治这个。先算「这一步的近似加速度」：

main.cpp · forwardStep()
TVerticesMat accelerationApprox;
if (strand.hasVelocitiesPrev)
{
    accelerationApprox = (strand.mVelocity - strand.mVelocitiesPrev) / params.dt;
    strand.hasApproxAcceleration = true;
}

FloatingType gravNorm = 10.f;
Vec3 gravDir;
gravDir << 0, -1, 0;

accelerationApprox = (v - vPrev)/dt 是用前后两帧速度差估出来的加速度 $\tilde{\mathbf a} \approx (\mathbf v^t - \mathbf v^{t-1})/h$。gravDir = (0,-1,0) 是重力方向，gravNorm = 10 是重力模长（和 gravity 一致）。只有当上一帧速度已存在（hasVelocitiesPrev，即非第一帧）时才有得算。然后逐顶点决定初始猜测：

main.cpp · forwardStep()
FloatingType accelerationComponent = 0.f;
if (strand.hasApproxAcceleration) {
    for (int iV = 0; iV < strand.numVerts; iV++)
    {
        accelerationComponent = accelerationApprox.col(iV).dot(gravDir);
        accelerationComponent = accelerationComponent < gravNorm ? accelerationComponent : gravNorm;
        accelerationComponent = accelerationComponent > 1e-5f ? accelerationComponent : 0.f;
        strand.mVertPos.col(iV) = strand.mVertPrevPos.col(iV) + params.dt * strand.mVelocitiesPrev.col(iV)
            + params.dt * params.dt * gravDir * accelerationComponent;
    }
}
else
{
        strand.mVertPos = strand.inertia;
}

核心是 accelerationComponent 这个标量，对每个顶点单独算，对应论文里的 $\tilde a$：

  1. = accelerationApprox.col(iV).dot(gravDir)：把该顶点的近似加速度投影到重力方向。一个真在自由下落的顶点，加速度 $\approx(0,-g,0)$，点乘 $(0,-1,0)$ 得 $\approx +g=10$；一个被托住、几乎不加速的顶点，投影 $\approx 0$。
  2. < gravNorm ? ... : gravNorm：上夹到 gravNorm=10（不允许比纯重力还猛）。
  3. > 1e-5f ? ... : 0.f：下夹到 0（负的或几乎为零，就当 0，别往上弹）。

于是 $\tilde a$ 被钳在 $[0, 10]$：自由下落的顶点 $\tilde a\approx 10$（约等于满重力），被托住的顶点 $\tilde a\approx 0$。最后那行就是论文 Eq.17 的初始猜测：

$$\mathbf x^{(0)}_i = \mathbf x^{t}_i + h\,\mathbf v^{t-1}_i + h^2\,\hat{\mathbf g}\,\tilde a_i$$

对照代码：mVertPrevPos 是 $\mathbf x^t$，mVelocitiesPrev 是 $\mathbf v^{t-1}$，gravDir*accelerationComponent 就是 $\hat{\mathbf g}\,\tilde a_i$。换句话说：该落的顶点，初始猜测就带上几乎完整的重力位移（接近 $\mathbf y$）；不该落的顶点，初始猜测几乎停在原地，避免把一摞静止物体初始化成互相穿插的自由下落态。第一帧没有历史，就老老实实回退到 mVertPos = inertia（即 $\mathbf x^{(0)}=\mathbf y$）。

手算两个顶点：自由下落 vs 被托住

把这个 clamp 算具体。同一帧里取两个顶点，用 $h=\tfrac1{60}$、$g=10$，先备好这一步的重力位移尺度：

$$h^2 g=\Big(\tfrac1{60}\Big)^2\times10=\frac{10}{3600}\approx 0.00278\ (\text{约 }2.78\,\text{mm})$$
这是为讲解构造的自洽示例

下面两个 $\tilde a$ 取值（$\approx10$ 与 $\approx0$）和 2.78 mm 是把 clamp 两端算给你看的示意数字，展示「该落的吃满、被托住的归零」这个趋势，不对应某真实帧的精确测量。

顶点 A：自由下落。它上一帧就在加速下坠，前后帧速度差给出 $\tilde{\mathbf a}\approx(0,-10,0)$，投影到 $\hat{\mathbf g}=(0,-1,0)$：$\tilde a=(0,-10,0)\cdot(0,-1,0)=+10$。落进 clamp 上界 $[0,10]$ 的顶，保留为 $10$。于是它的初始猜测带上几乎完整的重力位移：

$$h^2\,\hat{\mathbf g}\,\tilde a=\tfrac1{3600}\cdot(0,-1,0)\cdot10=(0,-0.00278,0)$$

——往下 2.78 mm，和「直接用惯性 $\mathbf y$」给出的位移几乎一样。该落的顶点，热启动就让它落。

顶点 B：被托住（叠在别的物体上、或被硬弹簧拽住，这一步几乎不加速）。前后帧速度差 $\approx0$，投影 $\tilde a=(0,\sim0,0)\cdot(0,-1,0)\approx 0$；落进 clamp 下界判断 > 1e-5 ? : 0，归 $0$。它的重力位移项：

$$h^2\,\hat{\mathbf g}\,\tilde a=\tfrac1{3600}\cdot(0,-1,0)\cdot0=(0,0,0)$$

——零位移，初始猜测就停在 $\mathbf x^t+h\mathbf v^{t-1}$（若它本来静止，就是原地）。对照朴素初始化 $\mathbf x^{(0)}=\mathbf y$ 会把它也往下扔 2.78 mm，逼得后面的迭代再费劲把它从穿插/过拉里拽回来。clamp 到 0 直接省掉了这趟冤枉路。

一个细节：这里用的是 vPrev，不是 v

留意末行积分用的是 mVelocitiesPrev（$\mathbf v^{t-1}$）而非刚加完重力的 mVelocity。直觉上这是把「上一帧的运动趋势」往前推一格，再单独按 $\tilde a$ 补重力——而 $\tilde a$ 本身已经从前后帧速度差里读出了「这个顶点最近到底有没有在加速下落」。两件事合起来，就是「沿用旧动量 + 只给真在下落的顶点补重力」。

从 PBD 看 为什么 XPBD 做不了这一手

XPBD 的初始化是被钉死的：必须 predict $\mathbf x^*=\mathbf x+h\mathbf v+h^2\mathbf f_{ext}/m$，因为它的 Lagrange multiplier $\lambda$ 推导依赖这个特定起点。VBD 不一样——它只是在最小化 $G(\mathbf x)$，从哪个 $\mathbf x^{(0)}$ 起步都行，只要每步让 $G$ 下降就保证收敛（1.2、3.2）。「不强制某种初始化」正是 VBD 能做自适应初始化的前提。给优化器一个更好的起点，就少迭代几次、也更不容易在第一步就穿插——这是把「最小化视角」换成实打实性能的又一处兑现。

把它接回 1.4 那场拔河：惯性碗的碗底是 $\mathbf y$。自适应初始化做的，是不把所有顶点都先扔到碗底，而是按「它最近是不是真在被重力主导」决定起点离碗底多近。被弹力/接触拉着的顶点（拔河里弹性那头赢的顶点），干脆先别动——这正是优化器最想要的热启动。

这一节你要带走的

三件事：(1) forwardStep() 先把重力积进速度、pin 住 0 号顶点、算出惯性 $\mathbf y=\mathbf x^t+h\mathbf v^t+h^2\mathbf g$（存进 inertia，整步不变）；(2) 它不止步于 $\mathbf x^{(0)}=\mathbf y$，而是逐顶点算一个钳在 $[0,10]$ 的 $\tilde a$——自由下落的取满重力、被托住的取 0——给出更聪明的初始猜测（论文 Eq.17）；(3) 这种自由来自 VBD「只需让 $G$ 下降、不强制初始化」的本性，是 XPBD 给不了的。

初始猜测就位。下一节是全书的中心：把 solve() 这一次 Gauss-Seidel sweep 逐行拆开，让每一行都对上 3.3 / 1.5 的公式。
