simulate() 主循环与可视化

零件齐了，这一节把它们串成整机。simulate() 就是那台时钟：一帧套一个 substep，substep 里 forwardStep() → 反复 solve() → updateVelocity() → 写盘。读完它，你脑子里 TinyVBD 的整条调用链就闭合了。

主循环：帧 → substep → 迭代

main.cpp · simulate()
void simulate() {
    saveOutputs();
    params.dt = params.dt / params.substeps;
    TVerticesMat prevIterPos;
    prevIterPos.resizeLike(strand.mVertPos);

    for (frameId = 1; frameId < params.numFrames; frameId++)
    {
        for (step = 0; step < params.substeps; step++)
        {
            forwardStep();
            float omega=1.f;
            for (iter = 0; iter < params.numIterations; iter++)
            {
                prevIterPos = strand.mVertPos;
                solve();

                if (params.useAcceleration)
                {
                    omega = getAcceleratorOmega(iter + 1, params.accelerationRho, omega);
                    applyAccelerator(omega);
                    strand.prevprevPos = prevIterPos;
                }
            }

            updateVelocity();

        }

        std::cout << "Frame: " << frameId << " finished!\n";
        saveOutputs();
    }
}

从外往里读这三层循环：

  • 开场两行：先 saveOutputs() 把初始帧（第 0 帧）写出去，再 params.dt /= params.substeps——把一帧的时间切成 substeps 份。默认 substeps=1，所以 dt 不变还是 $1/60$；但若你把 substeps 调成 10，每个 substep 的 $h$ 就变成 $1/600$（小 $h$ = 惯性碗更陡、更稳，代价是算 10 倍，见 5.3）。
  • 帧循环（frameId 从 1 到 numFrames-1，默认 300 帧）：每帧末尾 saveOutputs() 落一帧、打一行 Frame: N finished!。
  • substep 循环：每个 substep 是一次完整的隐式欧拉步——forwardStep()（4.2：算 $\mathbf y$、挑初始猜测）打头，updateVelocity() 收尾。
  • 迭代循环（numIterations 次，默认 100）：每次 prevIterPos = mVertPos 存档，再 solve() 做一遍 Gauss-Seidel sweep（4.3），把 $G$ 往下推一截。可选的 Chebyshev 加速（4.4）挂在每次 sweep 之后；默认 useAcceleration=false，这块不跑。

一句话：「迭代 100 次 solve」= 把这一 substep 的 $\arg\min G$ 解到够准；「substep」= 推进一个隐式欧拉时间步；「frame」= 攒够 substep 推进一整帧并写盘。VBD 的无条件稳定意味着即使把 numIterations 压到很低，它也不炸，只是欠收敛、偏软（0.1 那个 strand-lab 就让你拉这个滑块）。

updateVelocity()：从位移反算速度

main.cpp · updateVelocity()
void updateVelocity() {
    strand.mVelocitiesPrev = strand.mVelocity;
    strand.mVelocity = (strand.mVertPos - strand.mVertPrevPos) / params.dt;
    strand.mVelocity.col(0).setZero();

    strand.hasVelocitiesPrev = true;
}

一个 substep 的位置解完，速度就是位移除以时间步——这是隐式欧拉的定义 $\mathbf v^{t+1}=(\mathbf x^{t+1}-\mathbf x^t)/h$。逐行：

  • mVelocitiesPrev = mVelocity：先把旧速度 $\mathbf v^{t}$ 存一份——这正是 4.2 自适应初始化要用的 $\mathbf v^{t-1}$。
  • mVelocity = (mVertPos - mVertPrevPos)/dt：用这一步前后位置（mVertPrevPos 是 forwardStep() 存的 $\mathbf x^t$）反算新速度。
  • mVelocity.col(0).setZero()：再次 pin 住 0 号顶点（保险，确保挂点速度严格为 0）。
  • hasVelocitiesPrev = true：从此 mVelocitiesPrev 有效，下一帧的自适应初始化就能启用了——这就是为什么只有第一帧退回朴素的 mVertPos = inertia。

saveOutputs()：每帧一个 JSON

main.cpp · saveOutputs()
void saveOutputs() {
    std::vector<std::array<FloatingType, 3>> verts;

    for (int iV = 0; iV < strand.numVerts; ++iV)
    {
        auto v = strand.mVertPos.col(iV);
        std::array<FloatingType, 3> pt = { v[0], v[1], v[2] };
        verts.push_back(pt);
    }

    std::ostringstream aSs;
    aSs << std::setfill('0') << std::setw(8) << frameId;
    std::string outNumber = aSs.str();
    std::string outFile = outPath + "/A" + outNumber + ".json";
    nlohmann::json j;
    j["pos"] = verts;
    saveJson(outFile, j, 2);
}

它把当前所有顶点位置收成一个 list，写进 j["pos"]，文件名是 8 位补零的 A%08d.json（如 A00000001.json）。一帧一个文件，跑完默认 300 帧就得到 A00000000.json … A00000299.json 一整串。可视化交给 VisualizeStrand.blend——它在 Blender 里把这串 JSON 的 "pos" 读进来，转成 obj / 逐帧曲线，让你看到 strand 甩动。也就是说 TinyVBD 自己不画图，只吐数据，渲染解耦在 Blender 那头。

一个真实的小坑：outPath 是 Windows 路径，建文件夹只在 _WIN32 下有效

诚实提醒：各 initialize*() 里的 outPath 写死成 Windows 风格的 C:\Data\...（例如 "C:\\Data\\Test4_20Verts_30degree_withSkip_stiffness1e8"）；而 createFolder() 整个函数体包在 #ifdef _WIN32 里——非 Windows 平台它直接返回、不建任何目录，folderExists() 同理。所以在 Linux/macOS 上直接跑，要么写不出文件（目录不存在）、要么路径无意义。想在别处跑，去对应 initialize*() 把 params.outPath 改成一个你这边存在的目录（比如 "./out" 并先手动 mkdir），就能正常吐 JSON 了。算法本身和平台无关，只是这层 IO 没做跨平台。

把整条链连起来

退一步看全景，一次 simulate() 就是：

$$\underbrace{\text{frame}}_{\times 300}\;\to\;\underbrace{\text{substep}}_{\times 1}\;\to\;\Big[\;\texttt{forwardStep} \to \underbrace{\texttt{solve}}_{\times 100\;(+\text{Cheb?})} \to \texttt{updateVelocity}\;\Big]\;\to\;\texttt{saveOutputs}$$

每个箭头你都已经逐行读过：forwardStep()（4.2）备好惯性 $\mathbf y$ 和初始猜测；solve()（4.3）一遍遍把 $G$ 推向谷底；可选 Chebyshev（4.4）给它提速；updateVelocity() 反算速度、为下一步的自适应初始化留好 $\mathbf v^{t-1}$；saveOutputs() 落盘交给 Blender。这就是 TinyVBD 的全部——一根 strand、一套 Eigen、一个 min $G(\mathbf x)$，被这台三层时钟一帧帧地仿真出来。

这一节你要带走的

三件事：(1) simulate() 三层循环——帧（300）× substep（1，会把 $h$ 切细）× 迭代（100 次 solve），可选 Chebyshev 挂在每次 sweep 后；(2) updateVelocity() 用 $(\mathbf x-\mathbf x^t)/h$ 反算速度，先存旧速度供自适应初始化、再 pin 住 0 号顶点；(3) saveOutputs() 每帧写一个 A%08d.json 的 "pos"，由 VisualizeStrand.blend 可视化——但 outPath 是 Windows 路径、createFolder 只在 _WIN32 下有效，换平台要改。
