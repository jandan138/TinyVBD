一个最小的 Newton 程序：从 ModelBuilder 到 step 循环

前几节我们把 Newton 的 Warp 编程模型、Model/State 数据分离都拆开看了。这一节把它们拼回去：写出一个能真正跑起来的最小 VBD 程序。你会发现，它和你烂熟的 TinyVBD simulate() 是同一个骨架——只是搭场景、着色、循环的几行 API 换了个名字。我们逐行走一遍，并把每一行接回那根 strand。

先看全貌

下面是一段忠实于 Newton 真实 example（basic 系列里的 pendulum/cloth 程序）骨架的最小程序。pendulum example 是 rigid 的，这里我们换成 particle/cloth 的写法，但建场景 → 着色 → finalize → step 循环这条主干是所有 Newton 程序通用的：

minimal_vbd.py
import newton

# 1. 建场景：往 builder 里加内容
builder = newton.ModelBuilder()
# ... 加 particles / cloth 三角网格 / soft body / shapes ...
# builder.add_cloth_grid(...) 之类的 helper 把顶点、边、三角填进去

# 2. 着色：VBD 必需，且必须在 finalize 之前
builder.color()

# 3. 冻结成 Model（静态数据从此只读）
model = builder.finalize()

# 4. 造一个 VBD 求解器
solver = newton.solvers.SolverVBD(model)

# 5. 准备运行时缓冲
state_0 = model.state()
state_1 = model.state()
control = model.control()
contacts = model.contacts()

dt = 1.0 / 60.0
substeps = 1

# 6. 主循环：每帧若干 substep
for frame in range(num_frames):
    for _ in range(substeps):
        state_0.clear_forces()
        model.collide(state_0, contacts)
        solver.step(state_0, state_1, control, contacts, dt)
        state_0, state_1 = state_1, state_0   # swap

就这些。六块：建场景、着色、finalize、造 solver、备缓冲、跑循环。下面一块一块对着 TinyVBD 读。

builder：搭场景的脚手架

建场景
builder = newton.ModelBuilder()
# ... builder.add_cloth_grid(...) / add particles / shapes ...

ModelBuilder 是一个可变的脚手架：你往里加顶点、边、三角、刚体、关节、材料参数，它就是个在内存里慢慢长大的场景描述。Newton 提供一批 helper（如往里灌一张 cloth 三角网格、灌一堆 particle、灌一个 box shape），它们最终都落到同一组数组里——顶点位置 particle_q、质量 particle_mass、边/三角的连接关系、材料 tri_materials 等。这一步只是描述，还没有任何求解器、还没有冻结。

从 TinyVBD 看 builder ↔ 那几个 initializeXxx()

TinyVBD 里 initializeTilted() / initializeStiffRatio() 干的就是 ModelBuilder 的活：手填 numVerts、edges、每边 stiffness、末端重质量 m1。区别只是 TinyVBD 把数据直接写进 VBDStrand 的成员，而 Newton 让你通过 builder 的 helper 来填，最后统一冻结。一根 strand → 一张 cloth，填的内容多了，姿势没变。

builder.color()：VBD 的入场券

着色（必需）
builder.color()

这一行最容易被漏掉，却是 VBD 能并行的前提。graph coloring 给每个顶点染一种颜色，规则是同色顶点之间不共享任何 force element（不在同一条边/同一个三角上）。这样同一种颜色的所有顶点，它们的 3×3 牛顿步互不干扰，就能在 GPU 上一口气并行算完；而不同颜色之间仍按顺序扫，于是整体保留了 Gauss-Seidel 的「用最新值」收敛性。着色结果存进 model.particle_color_groups。

为什么必须在 finalize() 之前调？因为着色要写进 Model 的静态数据，而 finalize() 一旦把 Model 冻结成只读，就没机会再往里塞颜色分组了。换句话说：颜色必须先存在，才能被冻进 Model。SolverVBD 在构造时会去 Model 里找这组颜色——找不到就直接报错（见下）。

从 TinyVBD 看 这一步在 TinyVBD 里「不存在」

TinyVBD 是单线程 CPU，solve() 就一个 for iV 顺着扫，天然是串行 Gauss-Seidel，不需要也没有着色。builder.color() 正是「一根 strand 串行 → 一块 cloth 按颜色并行」这句主线里多出来的那一步：它不改变在解的那个 $\min G(\mathbf x)$，只是把「该按什么顺序、哪些顶点能同时动」这件事提前规划好。Part 8 会用专门的 widget 把顶点图着色拆给你看。

最容易踩的坑：忘了 builder.color()

SolverVBD 构造时会检查 model.particle_color_groups 是否为空；为空就直接抛 ValueError，提示你必须在 finalize 之前调用 ModelBuilder.color()（或 set_coloring()）。这是 VBD 特有的要求——别的求解器（如 XPBD）不需要着色，所以从 XPBD 例子改过来时极易漏掉。症状很直白：程序在 SolverVBD(model) 这一行就崩，而不是跑到一半才出错。记住顺序：加内容 → color() → finalize() → SolverVBD。

interactivebuilder.color() 在网格上到底干了什么
这就是 builder.color() 这一行背后的事：给每个顶点染色，使同色顶点之间不共享任何 force element（不在同一条边/三角上）。同色的一组可以一口气并行算 3×3 牛顿步，颜色之间再串行扫——既拿到并行、又保住 Gauss-Seidel 的「用最新值」。点一点看染色规则与每种颜色能并行的顶点集合；Part 8 还会用它把按色 sweep 拆得更细。

finalize 与 solver：从可变到只读

冻结 + 造 solver
model = builder.finalize()
solver = newton.solvers.SolverVBD(model)

builder.finalize() 把脚手架冻结成一个 Model：所有静态数据（顶点初始位置、质量、连接关系、材料、刚度、着色分组、重力）从此只读，整理成按 id 索引的 GPU 数组。newton.solvers.SolverVBD(model) 拿到这个 Model，分配好自己的工作缓冲（每顶点的力 / Hessian / 位移等），并把那组颜色读出来准备按色 sweep。对 particle 它走 VBD，对 rigid 它走 AVBD——本节只关心 particle 这条路径。

从 TinyVBD 看 Model 静态 ↔ VBDStrand 的不变量

TinyVBD 里 orgLengths、edgesStiffness、vertexMass、vertAdjacentEdges 这些一旦 from(...) 建好就不再变——它们就是 Newton 的 Model：仿真过程中只读的「这根东西长什么样」。会变的只有位置和速度，那是 State 的事。Newton 把这条「静态/动态分离」做成了显式的两个对象。

state / control / contacts：运行时缓冲

备缓冲
state_0 = model.state()
state_1 = model.state()
control = model.control()
contacts = model.contacts()

State 是某一时刻的动态快照：particle_q（位置）、particle_qd（速度）、particle_f（力）。我们造两个——state_0 当输入、state_1 当输出，因为 solver.step 不就地改写，而是「读 in、写 out」，算完再交换（double buffering）。model.control() 给一个 Control（控制输入，纯 particle 场景基本空着）；model.contacts() 给一个 Contacts 缓冲，等着碰撞检测往里填。这些都由 Model 出厂，形状和 Model 一致。

主循环：collide → step → swap

每 substep 三件事 + swap
for frame in range(num_frames):
    for _ in range(substeps):
        state_0.clear_forces()
        model.collide(state_0, contacts)
        solver.step(state_0, state_1, control, contacts, dt)
        state_0, state_1 = state_1, state_0

每个 substep 干四件事：(1) state_0.clear_forces() 把上一步残留的 particle_f 清零；(2) model.collide(state_0, contacts) 用当前位置跑碰撞检测，把接触对填进 contacts；(3) solver.step(state_0, state_1, control, contacts, dt) 解一个隐式欧拉步——内部就是 8.2 会讲的初始化 → 按色迭代 VBD → 由位移算速度三段，从 state_0 读、把新位置/速度写进 state_1；(4) 交换两个 state，让这一步的输出成为下一步的输入。dt 是 substep 的时间步（默认 $1/60$ 再除以 substeps）。

从 TinyVBD 看 这就是 simulate() 的逐行翻译

把 TinyVBD simulate() 摆旁边一眼就对上：它每帧每 substep 做 forwardStep()（前向积分 + pin 0 号）→ 迭代 numIterations 次 solve() → updateVelocity()，整段对应 Newton 一次 solver.step()——惯性初始化、按色（TinyVBD 是单 sweep）的牛顿迭代、后向差分算速度都被打包进去了。TinyVBD 用 saveOutputs() 手写一串 JSON（字段 pos）做可视化；Newton 这边对应的是把 state_0 交给 viewer 渲染或导出，不用你手撸 IO。还有 pinning：TinyVBD 在 forwardStep/updateVelocity 里把第 0 顶点速度强制清零钉死；Newton 对应的是在 builder 阶段给该 particle 打 particle_flags（标记为不可动），由 solver 自动跳过。一句话——循环还是那个循环，min G(x) 还是那个 min G(x)。

为什么是「两个 state 来回 swap」

这个 in/out 双缓冲不是 VBD 专属，而是 Newton solver.step(state_in, state_out, ...) 契约的统一形状（pendulum 那个 rigid 例子也是同样的 swap）。好处是 step 是个干净的纯函数式映射「旧状态 → 新状态」，便于把整段循环用 CUDA graph 录下来零开销重放。本节不展开 graph capture，知道循环长这样即可。

这一节你要带走的

一个能跑的 Newton VBD 程序就六步，记成一条流水线：ModelBuilder() 加内容 → color()（VBD 必需，且必须在 finalize 前）→ finalize() 冻结成只读 Model → SolverVBD(model) → 取 state_0/state_1、control、contacts → 循环里 clear_forces + collide + step + swap。它和你吃透的 TinyVBD simulate() 是同一个骨架，多出来的只有 color() 这一步——因为我们要从「一根 strand 串行」走向「一块 cloth 按颜色并行」。唯一最该刻进肌肉记忆的坑：漏 color() 会在造 solver 那行直接崩。
