Warp 编程模型：读 kernel 前要会的那点东西

Newton 的求解器不是用普通 Python 写的——它的每一行热点代码都是一段 Warp kernel，会被编译成 GPU 上的并行程序。要读懂后面那些 VBD kernel，你先得换一副眼镜：从「一根 strand 串行扫过去」切到「成千上万个 thread 同时各干一份」。这一节就把这副眼镜配给你。

为什么 Newton 不是普通 Python

上一节我们说 Newton 建立在 Warp 之上。Warp 是 NVIDIA 的一个 Python 库，但它干的事和 NumPy / PyTorch 很不一样：你用 Python 的语法写一个函数，加上一个装饰器，Warp 就把它 即时编译（JIT）成原生的 CUDA kernel，丢到 GPU 上成百上千个线程并行去跑。换句话说，你写的是 Python，跑的是 GPU 代码。

这件事对读代码的影响是根本性的。你在 TinyVBD 里读到的 solve() 是一个老老实实的 for 循环，CPU 一个顶点接一个顶点地处理。Newton 里对应的东西长得也像普通函数，但它不是被「调用」的，而是被「launch」的——你说一句「给我开 N 个线程」，这个函数体就会被 N 份拷贝同时执行，每份处理一个顶点。读 Newton 代码时若还用串行循环的脑子去想，几乎每个 kernel 都会读拧。

@wp.kernel 与 wp.launch：并行的入口

Warp 里最核心的两件事：用 @wp.kernel 标记一个「能被并行跑」的函数，用 wp.launch 真正把它铺开到 N 个线程上。

示意
@wp.kernel
def my_kernel(a: wp.array(dtype=wp.vec3), b: wp.array(dtype=wp.vec3)):
    i = wp.tid()          # 这个 thread 是第几号
    b[i] = a[i] * 2.0     # 它只负责第 i 份数据

wp.launch(my_kernel, dim=N, inputs=[a, b])   # 一口气开 N 个 thread

wp.launch(kernel, dim=N, inputs=[...]) 是唯一的发动机：dim=N 说「开 N 个线程」，inputs 是这些线程共享的那些数组。launch 之后，my_kernel 的函数体会被 N 个线程同时执行，彼此独立。这就是所谓的 SPMD（Single Program, Multiple Data）：同一段程序，跑在不同的数据上。

每个 thread 的视角：tid = wp.tid()

站在单个 thread 的视角看，kernel 体其实非常朴素。它进门第一件事通常就是问一句「我是谁」：

$$\texttt{i = wp.tid()}\quad\Longrightarrow\quad i \in \{0,1,\dots,N-1\}$$

wp.tid() 返回这个线程的全局编号。线程 0 拿到 i=0，线程 1 拿到 i=1，依此类推。拿到自己的 i 之后，整个 kernel 体就只围着「第 i 份数据」打转——读 a[i]、写 b[i]。它不知道、也不关心别的线程在干什么。每个 thread 都可以独立地走自己的 if 分支、独立地 return 提前退出，互不影响。把这句话刻进脑子，后面读 kernel 会顺很多：kernel 体描述的是「一个线程对一个元素做的事」，而不是「对整个数组做的事」。

写 PyTorch/NumPy 时，你想的是「整个数组乘 2」（b = a * 2），并行被藏在算子背后，你看不见线程。Warp 反过来：它把线程明确摆在你面前，你写的是「第 i 个元素乘 2」，再用 dim=N 把它复制 N 份。这更接近 CUDA 的写法，也更接近 GPU 真实在干的事。代价是你得自己想清楚「谁碰谁的数据」——这正是后面 graph coloring 要解决的问题。

@wp.func 与 Warp 的类型

kernel 体里往往会调用一些小的辅助函数，比如「给定一条边，算它的弹簧力和 Hessian」。这种被 kernel 调用的设备函数用 @wp.func 标记——你可以把它类比成 CUDA 里的 __device__ 函数：它不能被 wp.launch 直接铺开，但能在 kernel（或别的 @wp.func）里被调用。Newton 的 VBD 里那些 evaluate_*_force_hessian 就都是 @wp.func。

数据这边，Warp 的主力容器是 wp.array，它是一块 GPU 上的连续内存，声明时要带 dtype。常见的 dtype 正好对得上你写弹性力学要用的东西：

  • wp.float32 / wp.int32——标量；
  • wp.vec3——三维向量，顶点位置、速度、力都是它（wp.array(dtype=wp.vec3)）；
  • wp.mat33——3×3 矩阵，正好装每顶点那个 3×3 的 Hessian；
  • wp.struct——把一组字段打包成一个结构体，方便整团传进 kernel。Newton 里就有像 ParticleForceElementAdjacencyInfo 这样的 struct，把「每个顶点邻接哪些 force element」的索引信息打成一包。

注意这里的对应关系：TinyVBD 里你用 Eigen 的 Vec3 / Mat3 存顶点的力和 Hessian；Newton 里换成 wp.vec3 / wp.mat33，但数学是同一套，只是从 CPU 上的一个矩阵，变成 GPU 数组里第 i 个元素。

读一段真的：forward_step kernel

把上面的眼镜戴上，来看 Newton particle VBD 里一段真实代码——惯性初始化的 forward_step kernel。它干的事，就是 TinyVBD forwardStep() 里「前向积分、算预测位置」那一步：

particle_vbd_kernels.py
@wp.kernel
def forward_step(
    dt: float,
    gravity: wp.array[wp.vec3],
    pos_prev: wp.array[wp.vec3],
    pos: wp.array[wp.vec3],
    vel: wp.array[wp.vec3],
    inv_mass: wp.array[float],
    ...
    inertia_out: wp.array[wp.vec3],
):
    particle = wp.tid()

    pos_prev[particle] = pos[particle]
    ...
    vel_new = vel[particle] + (gravity[0] + external_force[particle] * inv_mass[particle]) * dt
    inertia = pos[particle] + vel_new * dt
    inertia_out[particle] = inertia

逐句对照你已经会的东西：第一行 particle = wp.tid()——「我负责哪个顶点」。然后 pos_prev[particle] = pos[particle] 存一下上一帧位置（对应 TinyVBD 的 mVertPrevPos）。再算 vel_new = vel + (gravity + ...) * dt，最后 inertia = pos + vel_new * dt——这就是预测位置 $\mathbf y \approx \mathbf x^t + h\mathbf v^t + h^2\mathbf a_{ext}$。整个 kernel 体只描述「一个顶点」的事；wp.launch(forward_step, dim=numParticles, ...) 一发，所有顶点的惯性位置就在 GPU 上一次性算完了。

一处要诚实点名的差异

这段 forward_step 用的是朴素惯性初始化：$\mathbf y = \mathbf x + (\mathbf v + \mathbf g\,dt)\,dt$，直接拿完整重力外推。它不是 TinyVBD 那套由 accelerationComponent 决定的自适应初始化（按上一帧加速度在重力方向的投影来缩放）。这是 Newton particle 路径和 TinyVBD 的真实差异之一，后面 Part 8 会专门对照。

从 TinyVBD 看 串行 for iV ↔ 一次 dim=numVerts 的 launch

TinyVBD 的 solve() 里有一个 for iV = 1 .. numVerts-1 的循环，CPU 一个顶点接一个顶点地解牛顿步。Newton 把这件事翻译成：一次 wp.launch，dim=numVerts，每个 thread 干一个顶点，全部并行。iV 这个循环变量，就变成了每个 thread 各自的 wp.tid()。

但别急着以为「串行循环 = 一次 launch」是免费的等价替换。TinyVBD 的串行循环天然是 Gauss-Seidel：解第 iV 个顶点时，前面 iV-1 个的新位置已经就位。一旦把所有顶点同时并行，相邻顶点会同时读写、抢同一条边的数据，Gauss-Seidel 的次序就乱了。Newton 解决它的办法是 graph coloring：把顶点分成几「色」，同色顶点互不相邻、可以安全地一次 launch 全并行，颜色之间再串行。这件事我们留到 8-5 专门讲——这里你只要记住：launch 给了你并行，但「能不能并行」要由 coloring 来许可。

最后一块拼图：CUDA graph capture

仿真主循环每帧要 launch 几十上百个 kernel，逐个从 CPU 提交也有开销。Warp 提供 wp.ScopedCapture：把一整段 kernel 序列「录制」成一张 CUDA graph，之后每帧用 wp.capture_launch 零开销地整段重放，省掉反复提交的成本。你读 Newton 的 example 时会看到它——现在知道是这么回事即可。

这一节你要带走的

三件事：(1) Newton 的热点代码是 Warp kernel，用 @wp.kernel 写、用 wp.launch(dim=N) 铺到 N 个线程上，是 SPMD 模型；(2) 读 kernel 要切到「单个 thread 视角」——i = wp.tid() 拿到自己的编号，只碰第 i 份数据，分支和提前退出都各走各的；(3) 数据用 wp.array（dtype 是 wp.vec3 / wp.mat33 这些），辅助逻辑用 @wp.func，整段循环可被 CUDA graph 录下来重放。
