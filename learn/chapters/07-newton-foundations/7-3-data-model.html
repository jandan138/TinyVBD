Newton 数据模型：Model / State / Control / Contacts

你已经看懂了 Warp 的 kernel / array / launch——那是「怎么算」。这一节讲「算什么」：Newton 把整个仿真世界拆成四个互不相同的对象，Model、State、Control、Contacts。TinyVBD 把位置、速度、惯性全塞进一个 VBDStrand 对象里；Newton 偏偏要把它们分门别类。搞清这套分法，后面读 solver、读主循环就有了骨架。

为什么要把世界拆成四块

TinyVBD 只有一根 strand，所有东西——顶点位置 mVertPos、速度 mVelocity、惯性目标 inertia、质量 vertexMass、边和刚度——都挂在同一个对象上，随手取用。这在 400 行里没问题。但 Newton 要同时跑成千上万的粒子、刚体、关节，还要支持可微、支持 CUDA graph 重放、支持把一份场景反复积分很多步。这时候「什么东西从不改、什么东西每步都变」就必须分清楚，否则没法谈缓存、没法谈双缓冲、没法谈微分。

所以 Newton 沿着一条朴素的轴线切开：静态的归 Model，动态的归 State。再把「外部控制输入」单拎出来叫 Control，把「碰撞检测的产物」单拎出来叫 Contacts。四个对象各管一摊，solver 在它们之间搬运数据。

Model：静态的世界

Model 是整个场景里从头到尾不变的那部分：有哪些粒子、哪些刚体（bodies）、哪些关节（joints）、哪些碰撞形状（shapes），以及它们的物理属性（props）。这些数组都按 id 索引。你会在 Model 上看到这些字段：

  • particle_q——粒子的初始位置（建模时的参考构型）；
  • particle_mass——每个粒子的质量，仿真过程中不变；
  • gravity——重力等全局物理常量；
  • particle_color_groups——图着色的结果（8.5 细讲），VBD 求解必需，也是只算一次、之后不动的静态数据。

记住一个判据：如果某个量在整个 simulate() 里都不会被写回，它就属于 Model。质量、刚度、拓扑、颜色分组都是这种「设定一次、查很多次」的东西。

State：某一时刻的快照

State 是仿真在某一个时刻的动态快照——会随时间一步步被改写的那部分自由度。对粒子（cloth / soft body 的顶点）来说：

  • particle_q: wp.array[wp.vec3]——当前位置；
  • particle_qd: wp.array[wp.vec3]——当前速度；
  • particle_f: wp.array[wp.vec3]——累加在粒子上的力。

对刚体，自由度不是一个 vec3 能装下的，所以用更大的类型：

  • body_q: wp.array[wp.transform]——刚体位姿，一个 transform 同时打包平移和旋转（四元数）；
  • body_qd: wp.array[wp.spatial_vector]——刚体的空间速度，一个 spatial_vector 同时打包角速度和线速度。

注意命名的对称性：q 是广义坐标（位置/位姿），qd 是它对时间的导数（速度）。粒子和刚体共用这套 q / qd 记号，只是底层类型不同。一个 State 对象就是「这一帧此刻，所有自由度长什么样」的完整描述。

从 TinyVBD 看 你那个 VBDStrand，被 Newton 拆开了

TinyVBD 把 mVertPos、mVelocity、inertia、vertexMass 全焊在一个 VBDStrand 对象里。Newton 做同一件事，只是把它们按「静/动」劈成两半：vertexMass 这种永不改的，去 Model（particle_mass）；mVertPos 去 State 的 particle_q，mVelocity 去 particle_qd。那个每步重算的惯性目标 inertia（$\mathbf y=\mathbf x+h\mathbf v+h^2\mathbf a$）则是 solver 在 step 内部临时算出来的中间量，不进 State 的公开字段——它只是通往新 particle_q 的踏脚石。同一份数据，TinyVBD 一个对象装下，Newton 按生命周期分了家。

Control：外界递进来的输入

Control 装的是「外部驱动信号」——不是仿真自己演化出来的，而是控制器、策略网络或脚本喂进来的。典型字段是关节相关的 joint_f（施加在关节上的力/力矩）和 joint_target_q（关节的目标角度，配合 PD 控制器把关节驱向目标）。对纯被动的 cloth strand，Control 多半是空的；但在 robotics 场景里，它就是 agent 把动作灌进物理世界的那个入口。Control 和 State 分开，正是因为它来自仿真之外。

Contacts：碰撞检测的产物

Contacts 装的是碰撞检测算出来的接触对——哪两个东西碰上了、接触点在哪、法向是什么。它不是你手填的，而是由 model.collide(state, contacts) 这一步根据当前 State 现场算出来再写进去的：

collide → 填充 contacts
model.collide(state_in, contacts)   # 读 state_in 的位置，写满 contacts

之所以每步都要重新 collide，是因为接触关系随位置变——这一帧贴着的两片布，下一帧可能就分开了。算好的 contacts 再交给 solver，让接触力进到每个顶点的 $\mathbf f_i$ / $\mathbf H_i$ 里。10.3 会讲 Newton 的 self-collision 怎么进 VBD 的牛顿步。

ModelBuilder：搭场景，然后 finalize 冻结

Model 既然是静态的，就需要一个「搭建阶段」把它一次性建好。这个角色是 ModelBuilder：你在它上面一点点加粒子、加 cloth、加形状、加关节，配好材料参数，最后调一次 builder.finalize()——它把所有零散设置打包、上传到 GPU、冻结成一个不可变的 Model。在 finalize 之前还可以改；finalize 之后 Model 就定型了，后面所有 step 都查它、但不改它。这条「build → finalize」的分界，正是「可变搭建」和「静态运行」的交接点。

从 Isaac / USD 看这套分法

这个「静/动分离」你其实早就熟。在 OpenUSD 里，stage 上的静态资产（mesh 拓扑、质量、材质）和每帧变化的动态属性（transform、velocity）本来就是两类东西——Newton 的 Model vs State 几乎是同一刀。在 Isaac 里，那张随时间刷新的 root-state tensor（位姿+速度的批量张量）对应的就是 Newton 的 State。一个差别值得留意：Newton 把 State 的双缓冲做成了显式的——它要你自己传 state_in 和 state_out 两个对象，而不是在一块 buffer 上原地改。下面就讲为什么。

求解器契约与双缓冲主循环

四个对象齐了，solver 的接口就很清爽。VBD 求解器的核心方法是：

solver step 契约
solver.step(state_in, state_out, control, contacts, dt)

读它的签名就能读出语义：从 state_in（旧快照）出发，结合 control 和 contacts，积分一个 dt，把结果写进 state_out（新快照）。输入和输出是两个不同的对象——这就是上面说的显式双缓冲（double-buffering）。Newton 不在一块内存上原地覆盖，而是 ping-pong：读这块、写那块，下一步再交换。这对可微（要保留旧值算梯度）和 CUDA graph 重放都更干净。

于是规范的主循环就长这样——collide → step → swap：

canonical step loop
state_0, state_1 = model.state(), model.state()
control, contacts = model.control(), model.contacts()
for frame in range(N):
    for _ in range(substeps):
        state_0.clear_forces()
        model.collide(state_0, contacts)               # 填充 contacts
        solver.step(state_0, state_1, control, contacts, dt)
        state_0, state_1 = state_1, state_0            # swap：新结果变成下一步的输入

逐行读：先 clear_forces() 把上一步残留的力清零；collide 按 state_0 的当前位置算出接触、写进 contacts；step 从 state_0 积一步到 state_1；最后那行 Python 元组交换把两个引用对调——刚算出的 state_1 成了下一轮的 state_0。这样一个 buffer 永远是「当前」、另一个永远是「下一步」，循环里它俩轮流坐庄。

从 TinyVBD 看 你的主循环其实也在做同一件事，只是省了一个 buffer

TinyVBD 的 simulate() 每个 substep 是 forwardStep() → 迭代若干次 solve() → updateVelocity()，全在 同一个 strand 对象上原地改。它不需要 state_in/state_out 两块，是因为它不可微、不录 CUDA graph、也只有一根 strand。Newton 把同样的「读旧态、算新态」拆成两个显式对象，换来的是可微和零开销重放的能力。核心节奏没变——都是「预测/初始化 → 解 → 推进速度」，只是 Newton 把「解」这步连同接触一起塞进了 solver.step，并要求你显式管好那两块 buffer。

interactive双缓冲 ping-pong：看 swap 到底换了什么
两块 State 缓冲 A / B：每步从 in 读、往 out 写、再 swap 指针——上一步的 out 变成这一步的 in。底部灰色的 Model 方块从不被写。把「explicit / in-place」开关切到 in-place，就退化成 TinyVBD 那样的单缓冲原地改，对照看为什么 Newton 要两块。
为什么非要两块 State，而不是原地改一块

单缓冲（原地改）省内存，TinyVBD 就是这么干的——它不可微、不录 graph、只有一根 strand，原地覆盖完全够用。Newton 坚持显式两块，是为了两件单缓冲做不到的事：(1) 可微——反向传播要算梯度，就得在新态写好后仍能读到旧态；原地覆盖会把旧值毁掉，梯度就断了。(2) CUDA graph 重放——把整段循环录成 graph 零开销重放，要求每步的读/写目标固定、不互相踩；in / out 分开正好满足。代价只是多一块 State 的显存，换来可微与零开销重放，对一个面向研究和 robotics 的引擎是划算的。

这一节你要带走的

记牢这张分工表：Model = 静态世界（拓扑、particle_mass、gravity、particle_color_groups，由 ModelBuilder.finalize() 冻结）；State = 动态快照（particle_q/particle_qd/particle_f，刚体的 body_q/body_qd）；Control = 外部输入（joint_f、joint_target_q）；Contacts = model.collide 的产物。solver 的契约是 solver.step(state_in, state_out, control, contacts, dt)，主循环是 collide → step → swap 的显式双缓冲。TinyVBD 把这一切焊进一个对象、原地改；Newton 按生命周期分了家，换来可微与 graph 重放。
