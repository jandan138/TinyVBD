Newton 是什么

你已经把那根 ~400 行的 strand 吃透了：min G(x)、per-vertex 的 $\Delta\mathbf x = \mathbf H^{-1}\mathbf f$、Gauss-Seidel 一个个扫过去。现在我们把镜头拉远——同样的数学，搬到一个真正给机器人和研究用的 GPU 引擎里，它叫 Newton。这一篇先认门：Newton 是谁、它和你熟的 Isaac / OpenUSD 是什么关系、为什么 VBD 会住在这里。

一句话：建在 Warp 之上的 GPU 物理引擎

翻开 Newton 的 README.md，第一句就把它定义清楚了：Newton is a GPU-accelerated physics simulation engine built upon NVIDIA Warp（README.md:8），面向 roboticists 和 simulation researchers。注意这里的两个关键词：GPU-accelerated 和 built upon Warp。你那根 strand 跑在 CPU 上、靠 Eigen 做 3×3 求逆；Newton 则把同一类计算搬到 GPU 上，而承载它的语言/框架就是下一节要讲的 Warp——NVIDIA 的一套用 Python 写、却编译成 GPU kernel 的库。

Newton 并不是凭空造的。它扩展并泛化了 Warp 里那个已经 deprecated 的 warp.sim 模块，并把 MuJoCo Warp 集成进来当 primary backend；同时强调 GPU 计算、OpenUSD 支持、differentiability（可微）和用户可扩展（README.md:10-11）。换句话说：warp.sim 是前身，Newton 是它的「正式版」继任者，而 MuJoCo Warp 提供了主力刚体后端。

谁做的：从三家公司到 Linux Foundation

Newton 由 Disney Research、Google DeepMind 和 NVIDIA 共同发起（README.md:15），现在是一个 Linux Foundation 项目，社区共建共维，代码以 Apache-2.0 开源（README.md:13）。对你这种从 EBench 背景过来的读者，这个出身值得记一笔：它不是某一家的内部玩具，而是一个开放治理、明确面向 robotics 与研究的公共底座——这也解释了为什么它要那么认真地支持 OpenUSD 和可微性。

它在生态里的位置可以这样摆：底层是 Warp（GPU 编程语言/运行时）；Newton 站在 Warp 之上做物理；再往上，像 Isaac Lab 这类机器人学习框架可以把 Newton 当作物理后端来驱动场景。资产与场景的描述则交给 OpenUSD——也就是你熟悉的那套 USD stage。

从 Isaac / USD 看：Model 与 State 的分离

Newton 的核心数据模型把世界拆成两半，这个划分你在 Isaac / USD 里其实见过。Model 是静态的那一半——粒子、刚体、关节、形状、物理属性，按 id 索引的一堆数组（particle_mass、gravity、particle_color_groups…），对应你心里 USD stage 上那些静态资产；State 是某一时刻的动态快照——particle_q（位置）、particle_qd（速度）、刚体的 body_q/body_qd，对应 Isaac 的 root-state tensor、USD 上随时间变的动态属性。一个先剧透的小坑：Newton/Warp 的四元数顺序是 (x, y, z, w)，而 Isaac 用 (w, x, y, z)——互转时要换序，附录会专门讲。

一个引擎，多个 solver

这里有个对理解 VBD 至关重要的事实：Newton 不是「一种解法」，而是一个装着多种 solver 的引擎。它同时提供 XPBD、VBD、MuJoCo、Featherstone、SemiImplicit 等多个求解器（见 docs/index.rst）。你在前六部里建立的整张谱系图——PBD / XPBD / Projective Dynamics / VBD 都在解同一个 $\min G(\mathbf x)$，只是「怎么解」不同——在 Newton 里几乎是字面成真的：它们就是并排摆着、可以替换的 solver。

我们这一篇要盯住的是其中的 SolverVBD。它的类 docstring 写得很清楚：对 particle 用 VBD，对 rigid 用 AVBD（Augmented VBD）（solver_vbd.py:88）。也就是说，你吃透的那套顶点级 block coordinate descent，正是这里 particle 路径的内核；而 rigid 那条路用的是 VBD 的增广拉格朗日升级版 AVBD——Part 10 会专门讲。这就是为什么 VBD/AVBD 住在 Newton 里：它们是这个多-solver 引擎里专门负责可形变体与刚体约束的那两块。

从 TinyVBD 看 你的 simulate() 主循环，在 Newton 里叫 solver.step()

回想你那根 strand 的 simulate()：每帧 forwardStep() 做惯性初始化，然后迭代若干次 solve() 扫顶点，最后 updateVelocity()。Newton 把这套循环收进了一个统一的求解器契约：solver.step(state_in, state_out, control, contacts, dt)（solver_vbd.py:1578）。state_in 是这一刻的快照，state_out 接住下一刻，control 是控制输入、contacts 是碰撞检测结果、dt 还是你那个 dt。一次 step() 内部，就是「初始化 → 按颜色迭代 → 收尾算速度」——和你的 simulate() 一一对应，只是搬上了 GPU。这里只是预告，后面几节会把每一段拆开对照着读。

主线没变，变的是并行方式

把上面这些串起来，本篇的主线其实只有一句话，后面会反复用到：min G(x) 没变；变的是「一根 strand 串行 → 一块 cloth 在 GPU 上按颜色并行」。你已经懂的所有数学——增量势能、per-vertex 牛顿步、Gauss-Seidel 单调下降——在 Newton 里一个都没丢。新增的全是「怎么把它高效地铺到成千上万个 GPU thread 上」的工程学：Warp 的编程模型、graph coloring、CUDA graph capture。接下来几节就沿着这条线一步步走。

这一节你要带走的

就四件事：(1) Newton 是建在 NVIDIA Warp 之上、面向 robotics 与研究的 GPU 物理引擎，扩展自已 deprecated 的 warp.sim、以 MuJoCo Warp 为主力后端（README.md:8,10-11）；(2) 它由 Disney Research / Google DeepMind / NVIDIA 发起，现为 Apache-2.0 的 Linux Foundation 项目；(3) 它是个多-solver 引擎（XPBD / VBD / MuJoCo / Featherstone…），其中 SolverVBD 对 particle 用 VBD、对 rigid 用 AVBD；(4) 你的 simulate() 在这里被统一成 solver.step(state_in, state_out, control, contacts, dt) 契约。
