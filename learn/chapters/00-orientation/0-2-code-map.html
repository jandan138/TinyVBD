代码地图：7 个文件，一次 simulate()

在钻进任何一行代码之前，先花五分钟拿到一张地图。TinyVBD 一共就 7 个文件，真正的算法集中在两个里；一次完整的仿真，骨架也只有「初始化 → 逐帧推进」两层循环。看清这张图，后面读代码就不会迷路。

七个文件，各管一摊

整个工程没有构建系统的层层嵌套，也没有第三方框架——唯一的外部依赖是 Eigen3（纯头文件的线性代数库）。下面这张表就是全部家当，按「你会花多少时间读它」从多到少排列：

文件是什么你要花多少注意力
main.cpp 仿真器本体。装着 StrandSim 结构：所有初始化配置（initializeTilted 等）、forwardStep()、solve()、Chebyshev 加速、simulate() 主循环、以及把结果写成 JSON 的 saveOutputs()。 几乎全部。这本书 Part 4 会把它逐行读完。
Strand.h 数据结构。Mesh 基类持有顶点位置 / 速度 / 质量；VBDStrand 在它之上加 edges、edgesStiffness、vertAdjacentEdges、orgLengths、inertia 等，并用两个 from(...) 把一根 strand 装配出来。 很多。4.1 专讲。
Types.h 类型别名与宏。FloatingType=float、一堆 Eigen typedef（Vec3、Mat3、TVerticesMat=3×N 顶点矩阵…）、SQR/CUBE 宏、GRAVITY_AXIS=1（y 轴向上）、以及 RELEASE_ASSERT。 扫一眼即可，需要时回查。
json.hpp nlohmann 的单头文件 JSON 库。纯粹用来把每帧顶点位置序列化到磁盘，和算法毫无关系。 不用读，当黑盒。
VisualizeStrand.blend 一个 Blender 工程。把输出的 JSON 序列转成 obj 网格，让你能在 3D 里看这根 strand 怎么甩。 看结果时才用，4.5 会讲。
CMakeLists.txt 构建脚本。指明唯一依赖 Eigen3，生成可执行文件。 编译时用一次。
README.md 一页纸说明：这是 SIGGRAPH 2024 论文 Vertex Block Descent 的极简 demo，默认跑 high mass ratio 实验，输出 JSON 给 blend 可视化。 读一遍。
真正的算法只藏在两个文件里

把 json.hpp（IO 黑盒）和那些辅助文件刨掉，你会发现整套 Vertex Block Descent 其实就活在 main.cpp + Strand.h 里，加起来不过几百行。这正是 TinyVBD 的价值：论文完整版要在 GPU 上跑上亿自由度，而这里把同一套数学剥到只剩骨架，让你能一眼看穿。

一次 simulate() 从头到尾发生了什么

main() 干的事简单到近乎潦草：造一个 StrandSim，调 initialize()，再调 simulate()。难点全在 simulate() 这一个调用里。我们先在高空俯瞰它的调用链，不抠细节：

$$ \texttt{initialize()} \;\rightarrow\; \underbrace{\texttt{simulate()}}_{\text{逐帧}}:\quad \texttt{forwardStep()} \;\rightarrow\; \big(\underbrace{\texttt{solve()}}_{\times\,\text{numIterations}}\big) \;\rightarrow\; \texttt{updateVelocity()} \;\rightarrow\; \texttt{saveOutputs()} $$

逐层拆开来读，每一层都对应后面一整节：

  • initialize()：选一个实验配置（默认 initializeTilted()，即 20 顶点、末端质量 1000× 的 high mass ratio 实验），把 strand 的顶点、边、刚度、rest length 全部装配好，并建好输出文件夹。
  • forwardStep()（每帧一次，4.2）：施加重力、pin 住第 0 个顶点、算出惯性预测位置 inertia $\mathbf y$，再给这一帧的求解定一个聪明的初始猜测（自适应初始化）。这一步把「这帧要解的最小化问题」摆好。
  • solve()（每帧重复 numIterations 次，4.3）：一次完整的 Gauss-Seidel sweep——从顶点 1 扫到末尾，每个顶点解一个 3×3 的牛顿步、就地更新自己的位置。这就是 VBD 的心脏。可选地，每次 sweep 后做一次 Chebyshev 加速（4.4）。
  • updateVelocity()（每帧一次）：位置已经收敛，用 $(\mathbf x-\mathbf x^{t})/h$ 反推出新速度，再 pin 住 0 号顶点的速度。
  • saveOutputs()：把这一帧的顶点位置写成一个 JSON 文件（A00000001.json…，字段 pos）。

一次 simulate() 的执行 trace

抽象的「帧 → substep → 迭代」三层结构容易看过就忘。把它代上默认参数（substeps=1、numIterations=100）走一遍具体计数，就成了一次「真实运行」的流水账——纵向是三层循环变量的取值，横向是「此刻调什么 × 改了哪个状态」：

framestepiter此刻调什么改了哪个状态
10—forwardStep()施重力 / pin 0 号 / 算 inertia $\mathbf y$ / 定初始猜测
100solve()（第 1 遍 sweep）顶点 1…19 各挪一个 $\Delta\mathbf x$
101…98solve() 再扫 98 遍$G$ 一遍遍下降
1099solve()（第 100 遍，末遍）位置基本收敛到 $\arg\min G$
10—updateVelocity()由 $(\mathbf x-\mathbf x^{t})/h$ 反推 $\mathbf v$
1——saveOutputs()写 A00000001.json
2…29900…99整套重复每帧一个 JSON，直到 299
把「100 次迭代」放到全程尺度

默认 numFrames=300 × substeps=1 × numIterations=100，而 frame 0 只 saveOutputs() 不解算，所以一次完整运行共调 solve() 约 $299\times1\times100 = 29900$ 次。「每帧 100 次迭代」听着不多，落到全程其实是近三万次 sweep——这就是 VBD「小步多迭」的尺度感（数字为默认参数下的自洽示例）。

从 GAMES103 看 这个外壳你太熟了

「每帧：施加外力 → 解出新位置 → 更新速度 → 输出」——这就是你写过无数遍的时间积分主循环骨架，forwardStep / updateVelocity 对应的正是 GAMES103 里的 predict 与 velocity update。唯一被换掉的，是中间「解出新位置」那一块：别人用一次大线性求解或一堆约束投影，VBD 用的是「重复 numIterations 次、每次扫一遍所有顶点」的 block coordinate descent。整本书的重头戏，就是这个被换掉的内核。

主循环的真实骨架

下面是 simulate() 去掉调试输出后的骨架，直接来自 main.cpp。注意它就是两层 for 循环：外层数帧，内层是 substep；每个 substep 里再嵌一层 numIterations 的迭代。读到这里你不需要懂每一行，只要看清这副「帧 → substep → 迭代」的三层结构：

main.cpp · simulate()
void simulate() {
    saveOutputs();
    params.dt = params.dt / params.substeps;
    TVerticesMat prevIterPos;
    prevIterPos.resizeLike(strand.mVertPos);

    for (frameId = 1; frameId < params.numFrames; frameId++)
    {
        for (step = 0; step < params.substeps; step++)
        {
            forwardStep();                 // 惯性 + 自适应初始化
            float omega = 1.f;
            for (iter = 0; iter < params.numIterations; iter++)
            {
                prevIterPos = strand.mVertPos;
                solve();                   // 一次 Gauss-Seidel sweep

                if (params.useAcceleration)
                {
                    omega = getAcceleratorOmega(iter + 1, params.accelerationRho, omega);
                    applyAccelerator(omega);          // Chebyshev 加速
                    strand.prevprevPos = prevIterPos;
                }
            }
            updateVelocity();              // x 反推 v
        }
        saveOutputs();                     // 写这一帧的 JSON
    }
}
默认参数（来自 SimulatorParams）

默认 numFrames=300、substeps=1、numIterations=100、dt=1/60、gravity=(0,-10,0)、useAcceleration=false。也就是说默认不开 substep、不开 Chebyshev 加速，每帧老老实实做 100 次完整 sweep。这些数字后面调实验时（Part 5）都会被你亲手改动。

一个可以诚实指出的小坑：输出路径

代码里 outPath 写死成 Windows 风格的 C:\Data\...，而建文件夹的 createFolder 只在 _WIN32 下真正生效。换到 Linux / macOS 上跑，你得先把这个路径改成本地能写的目录，否则 JSON 写不出来。这跟算法无关，纯粹是 demo 的工程细节。

这一节你要带走的

三件事：(1) 七个文件里真正承载算法的只有 main.cpp 和 Strand.h，其余是 IO、可视化和构建的配角；(2) 一次 simulate() 是「帧 → substep → 迭代」三层循环，每帧走 forwardStep → 多次 solve → updateVelocity → saveOutputs；(3) 这副外壳就是经典的时间积分主循环，被 VBD 替换掉的只有「解新位置」那一块。

这只是高空俯瞰——每个函数到底在算什么、为什么这么算，Part 4 会带你逐行读透。但在读代码之前，下一节先约定一下「这本书怎么读」：两座反复出现的桥，以及一些贯穿全书的小约定。
