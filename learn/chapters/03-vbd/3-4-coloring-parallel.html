Gauss-Seidel、Jacobi 与 graph coloring

到目前为止，我们的扫描都是顺序的：一个顶点更新完，下一个才动。这在一根 20 顶点的 strand 上无所谓，但在百万顶点的 GPU 场景里，「严格顺序」就是性能杀手。这一节讲 VBD 怎么既要 Gauss-Seidel 的快收敛、又要并行——靠的是给顶点图着色。

Gauss-Seidel vs Jacobi：一对老冤家

更新一遍所有顶点，有两种读法：

Gauss-SeidelJacobi
算 $\mathbf f_i,\mathbf H_i$ 用的邻居位置这一遍里最新的（已更新的用新值）全部来自同一份旧位置
更新方式就地，逐个先全算完，再统一写回
并行难（有顺序依赖）天然并行（互不依赖）
收敛快慢（信息一遍只传一格）
顺序相关是（扫描次序影响结果）否

取舍很清楚：Gauss-Seidel 收敛快但难并行——它要求处理顶点 $i$ 时邻居已经是新值，这就强行规定了先后；Jacobi 全用旧值、互不打扰，可以一口气并行算完所有顶点，但信息每遍只向外传播一格，收敛慢。GPU 想要 Jacobi 的并行，物理想要 Gauss-Seidel 的快收敛。VBD 想两个都要。

从 GAMES103 看 你迭代解线性系统时见过这对名字

解 $A\mathbf x=\mathbf b$ 时，Gauss-Seidel 把刚算出的分量立刻代回去用，Jacobi 则等一整轮算完再更新——课上通常说 Gauss-Seidel 收敛快约一倍但难并行。VBD 把同样的取舍搬到了非线性的逐顶点优化上，结论一致：顺序的快、并行的慢。下面的着色技巧，就是经典数值线性代数里「red-black ordering / multicolor Gauss-Seidel」那一招在仿真里的化身。

关键招：给顶点图着色

VBD 的妙处在于发现：Gauss-Seidel 之所以要顺序，只是因为相邻的两个顶点会互相影响——如果两个顶点不相邻（没有任何 force element 直接连着它们），那同时更新它们完全没有冲突，因为各自的 $\mathbf f_i$、$\mathbf H_i$ 都不依赖对方。

于是：给顶点图做 graph coloring——给每个顶点染一种颜色，要求任何一条 force element 的两个端点颜色不同。这样一来，同色的顶点之间一定互不相邻，可以安全地并行（同色内部是 Jacobi：彼此用旧值，但反正彼此无关）；而不同颜色之间按顺序处理，颜色与颜色之间就构成了 Gauss-Seidel（后一个颜色读得到前一个颜色刚更新的新值）。一句话：

同色 Jacobi 并行，跨色 Gauss-Seidel 顺序

把顶点按颜色分组。处理颜色 1（一整批顶点并行更新）→ 处理颜色 2（并行，但用得上颜色 1 的新值）→ … 一遍 sweep = 走完所有颜色。颜色数 = 一遍 sweep 里的顺序阶段数。颜色越少，并行度越高、同步开销越小。所以「图能用几种颜色染好」直接决定了 VBD 在 GPU 上跑多快。

为什么顶点着色比约束着色省颜色

这里有一个 VBD 相对 PBD/XPBD 的结构性便宜。PBD/XPBD 是约束（对偶）求解器，它们要并行就得给约束图 / dual graph 着色——保证同色的约束不共享顶点。而 VBD 是顶点（primal）求解器，它给顶点图着色。

顶点图通常比约束/对偶图稀疏得多：一个顶点接的边数（度数），往往远小于一条约束周围纠缠的其它约束数。图越稀疏，需要的颜色越少。论文报告，对同样的网格，VBD 的顶点着色所需颜色数，比 XPBD 的约束/对偶图着色少约一个数量级。颜色少意味着一遍 sweep 的顺序阶段更少、GPU 同步更少、并行批次更大——这是 VBD 能在 GPU 上高效跑大场景的工程基础之一。

从 PBD 看 同样要着色，但着的是不同的图

你如果在 GPU 上写过并行 XPBD，应该对「约束着色」不陌生：得把弹簧/约束分成若干批，同一批里没有两条约束共用顶点，才能并行投影。VBD 把着色对象从「约束」换成了「顶点」，而顶点图更稀疏，所以颜色数少了一截。再加上 VBD 是 primal、不按逆质量分配修正，它在高质量比下也不会像 XPBD 那样崩（5.1）。同样是「Gauss-Seidel + 着色并行」的骨架，换了内核就换了脾性。

说句实话：TinyVBD 自己没着色

TinyVBD 是单线程 CPU，做的是朴素顺序 Gauss-Seidel

别被上面这套并行叙事误导了去代码里找着色逻辑——找不到。TinyVBD 是个极简的单线程 CPU demo，它的 solve() 就是从 iV=1 一路顺序扫到末端的纯 Gauss-Seidel，没有任何 graph coloring、没有 GPU、没有并行批次。着色是完整版 GPU VBD 用来把这套顺序算法并行化的手段；在一根 20 顶点的 strand 上，顺序扫一遍快得很，根本不需要。这里讲它，是为了让你读懂论文里的并行部分、以及理解 VBD 为何能 scale 到百万顶点——而不是因为这份代码里有它。

这其实也是 TinyVBD 的价值所在：它把会让人分心的并行工程全剥掉了，只留下算法的数学骨架。等你彻底吃透了「逐顶点牛顿步 + 单调下降」，再回头看完整 VBD 的着色并行，就只是「把这套骨架铺到 GPU 上」的工程细节而已。

这一节你要带走的

三句话：(1) Gauss-Seidel 用最新值、快但顺序，Jacobi 用旧值、慢但并行；(2) 给顶点图着色——同色顶点互不相邻、可并行（同色 Jacobi），跨色按序（Gauss-Seidel）——VBD 因此兼得快收敛与并行，且顶点着色比约束/对偶着色省约一个数量级的颜色；(3) TinyVBD 本身是单线程 CPU、朴素顺序 Gauss-Seidel，并不着色，着色是完整 GPU 版本的并行手段。
