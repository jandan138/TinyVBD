particle 的惯性初始化：Newton 走的是朴素那条路

上一节我们看了 8.2 里 step() 的三段式，其中第一段 Initialize 负责把每个顶点往前推一步、给迭代挑个初始猜测。读 TinyVBD 时你已经吃透过这件事：4.2 的 forwardStep() 玩了一手漂亮的adaptive initialization。那么 Newton 的 particle 路径也这么聪明吗？答案是——没有。这一节就是要诚实地把这处差异摆到台面上。

forward_step kernel：三行就讲完

Newton 的 particle 前向积分就在一个叫 forward_step 的 Warp kernel 里，每个 thread 负责一个 particle。把和初始化直接相关的几行拎出来：

particle_vbd_kernels.py
@wp.kernel
def forward_step(dt, gravity, pos_prev, pos, vel, inv_mass,
                 external_force, particle_flags, inertia_out, displacements_out):
    particle = wp.tid()

    pos_prev[particle] = pos[particle]
    if not particle_flags[particle] & ParticleFlags.ACTIVE or inv_mass[particle] == 0:
        inertia_out[particle] = pos_prev[particle]
        return
    vel_new = vel[particle] + (gravity[0] + external_force[particle] * inv_mass[particle]) * dt
    inertia = pos[particle] + vel_new * dt
    inertia_out[particle] = inertia

逐行读，你会发现它和 TinyVBD 的前半截几乎一模一样：

  • pos_prev[particle] = pos[particle]：把这一步开始时的位置 $\mathbf x^t$ 存一份，留给末尾反算速度——和 TinyVBD 的 mVertPrevPos = mVertPos 完全对应。
  • particle_flags & ParticleFlags.ACTIVE 与 inv_mass == 0 这道闸：非激活或者「无穷重」（固定 pin 住）的顶点，惯性位置就锁回原地、不再往下算。这就是 Newton 版的「pin 第 0 个顶点」，只不过用 flag + 逆质量来表达，比 TinyVBD 写死 col(0) 更通用。
  • vel_new = vel + (gravity + external_force * inv_mass) * dt：把外力（重力 + 外部力按逆质量折成加速度）先积进速度，得到 $\mathbf v^{t+1}_{\text{pred}}$。注意外部力要乘 inv_mass 才变成加速度，而重力本身已经是加速度，直接加。
  • inertia = pos + vel_new * dt：这就是惯性预测位置 $\mathbf y$，写进 inertia_out，整步不再变。

把后两行合起来展开，得到的正是朴素惯性位置：

$$\mathbf y = \mathbf x + (\mathbf v + \mathbf g\,\Delta t)\,\Delta t = \mathbf x + \mathbf v\,\Delta t + \mathbf g\,\Delta t^2$$

这和全书第一节 1.2 给出的惯性项 $\mathbf y = \mathbf x^t + h\mathbf v^t + h^2 M^{-1}\mathbf f_{ext}$ 严丝合缝。注意——故事到这里就停了。inertia_out[particle] = inertia 写完，初始化就结束了。Newton 的迭代直接从这个 $\mathbf y$ 出发。

inertia 是「碗底」，不是「起点」那么简单

和 TinyVBD 一样，inertia（即 $\mathbf y$）算完后整步不动，会被后面 8.4 的 solve_elasticity 反复读取：惯性力 $\mathbf f \mathrel{+}= m(\mathbf y - \mathbf x)/\Delta t^2$ 拿当前 $\mathbf x$ 和这个固定 $\mathbf y$ 比。它既是惯性碗的碗底，也顺手当了迭代的初始位置——因为 Newton 这里没有再为初始猜测做任何额外加工。

少了什么：那个 accelerationComponent 投影

回忆 TinyVBD 4.2 在算完 inertia 之后还做了一整段：用前后两帧速度差估出近似加速度，把它投影到重力方向得到一个标量 accelerationComponent（论文 Eq.17 里的 $\tilde a$），钳在 $[0, g]$，然后逐顶点用

$$\mathbf x^{(0)}_i = \mathbf x^{t}_i + h\,\mathbf v^{t-1}_i + h^2\,\hat{\mathbf g}\,\tilde a_i$$

当初始猜测。它的精髓是区别对待：真在自由下落的顶点 $\tilde a\approx 1$（吃满重力位移、初始猜测接近 $\mathbf y$），被弹簧或接触托住的顶点 $\tilde a\approx 0$（几乎停在原地，不被先扔到碗底）。这样叠放、静止、被支撑的顶点不会一开始就被初始化成互相穿插的自由下落态。

Newton 的 forward_step 里，这一整段不存在。没有 accelerationApprox、没有 dot(gravDir) 投影、没有逐顶点的 $\tilde a$ 钳制。每个激活顶点都吃满一个 $\mathbf g\,\Delta t^2$ 的重力位移，无差别地落到惯性位置 $\mathbf y$。换句话说：

点名这处差异

Newton 的 particle 路径用的是朴素惯性初始化 $\mathbf y = \mathbf x + (\mathbf v + \mathbf g\,\Delta t)\,\Delta t$，不是 TinyVBD 的自适应初始化。代码里没有 accelerationComponent 那套逻辑——这正是 TinyVBD 的 forwardStep() 里那段被注释掉的 // mVertPos = inertia「baseline」，在 Newton 这边成了实际选用的做法。

这「先挪一下」具体多远？一个被托住的顶点

抽象说「先往下挪 $\mathbf g\,\Delta t^2$」不够痛——代个数看它到底多大。取一个静止、被下方弹簧或地面托住的顶点：它本该这一步几乎不动（支撑力正好顶住重力）。$\Delta t=\tfrac{1}{60}$、$g=9.8\,\text{m/s}^2$，两种初始化各把它挪多远？

$$\text{朴素：}\ \Delta y_{\text{naive}}=g\,\Delta t^2=9.8\times\Big(\tfrac{1}{60}\Big)^2=\frac{9.8}{3600}\approx2.7\times10^{-3}\,\text{m}=2.7\,\text{mm}$$

朴素初始化不管三七二十一，把这个静止顶点先下坠 2.7mm——它被无差别当成自由下落处理。自适应初始化则不同：它先估出这个顶点的近似加速度、投影到重力方向得到标量 $\tilde a$，再钳进 $[0,g]$：

$$\tilde a=\operatorname{clamp}\big(\hat{\mathbf g}\cdot\tilde{\mathbf a},\;0,\;g\big),\qquad \Delta y_{\text{adaptive}}=\Delta t^2\,\tilde a$$

对一个被托住、上一帧几乎没在加速下落的顶点，估出的 $\hat{\mathbf g}\cdot\tilde{\mathbf a}\approx0$，钳完 $\tilde a\approx0$，于是 $\Delta y_{\text{adaptive}}=\Delta t^2\times0=0$——原地不动。同一个顶点，朴素挪 2.7mm、自适应挪 0。这 2.7mm 不是错（迭代会把它拉回），但它是求解器开局就背上的一截「债」：弹性力/接触力得多花几次迭代把这个静止顶点从 $-2.7\text{mm}$ 拽回原位。在一整片叠放/被支撑的静止区域里，这种债逐顶点累积，热启动质量的差距就出来了。而真在自由下落的顶点，$\hat{\mathbf g}\cdot\tilde{\mathbf a}\approx g$、钳完 $\tilde a\approx g$，$\Delta y_{\text{adaptive}}\approx g\,\Delta t^2$——和朴素一致、吃满那 2.7mm。自适应的精髓正是这个区别对待：该落的落满、该停的真停。

2.7mm 是自洽示例，量级随 g、Δt 缩放

$2.7\,\text{mm}$ 是拿 $g=9.8$、$\Delta t=\tfrac{1}{60}$ 算出的自洽数，用来把「先挪一下」变成一个能掂量的长度。它随 $g$、$\Delta t^2$ 缩放（substep 越细这截下坠越小）。重点不是这个具体毫米数，而是朴素对所有顶点一视同仁地挪满 $g\,\Delta t^2$，自适应靠 $\tilde a$ 钳制让被支撑的顶点挪 0——这正是两条路在静止/接触场景下的可量化分岔。

从 TinyVBD 看 forwardStep() 的自适应分支，在 Newton 里没有对应物

TinyVBD 的 forwardStep() 有一个 if (strand.hasApproxAcceleration) 分支：有历史速度时走自适应初始化（逐顶点算 $\tilde a$、按投影补重力），第一帧才回退到 mVertPos = inertia。Newton 的 forward_step 等于永远走那个回退分支——不管第几帧，都直接 inertia_out = pos + vel_new * dt 收尾。两边的惯性 $\mathbf y$ 算法一字不差（位置积分逻辑相同），分岔只在「要不要再为初始猜测做自适应加工」这一步：TinyVBD 做了，Newton 没做。

这处差异要不要紧？诚实地说

朴素初始化最大的好处是简单：少一段逐顶点的投影与钳制、少存一份上一帧速度、kernel 更短，逻辑也更容易并行展开。对很多场景这完全够用——尤其是顶点本来就大多在自由运动、没有大片被支撑的静止区域时，$\mathbf y$ 已经是相当好的起点，自适应初始化省下的那几次迭代未必显眼。

代价也得说清楚：在叠放 / 接触 / 被强支撑的场景里，朴素初始化会把那些本该几乎不动的顶点先往下挪 $\mathbf g\,\Delta t^2$，于是迭代的第一步就引入了一截需要被弹性力和接触力拉回来的位移。这意味着求解器要多花几次迭代把它们「拉回」，热启动的质量不如自适应初始化好。4.2 强调过的那个好处——「不把所有顶点都先扔到碗底」——Newton 的 particle 路径在这里放弃了。

这里不必过度渲染：它不是 bug，也不会让结果错（迭代仍会收敛到同一个 $\arg\min G$，惯性初始化和自适应初始化只差一个起点，不改变目标函数）。它是一个务实的取舍——用初始猜测的质量换实现的简洁。要不要补上自适应初始化，取决于你的场景里「被支撑的静止顶点」占多大比例。把它和 8.1 反复说的那句主线对上：min G(x) 没变，变的只是工程实现上的一处选择。

和「没有 Chebyshev」是同一类诚实点名

这是 Newton particle 路径与 TinyVBD 的两处真实差异之一。另一处——Newton 的 particle solve 没有 Chebyshev 加速——留到 8.4 看 per-vertex Newton step 时再点。两处都遵循同一条原则：把工程实现上的简化老老实实摆出来，不假装 Newton 把 TinyVBD 的每个聪明小技巧都搬了过来。

这一节你要带走的

三件事：(1) Newton 的 forward_step kernel 做 pos_prev = pos、vel_new = vel + (gravity + external_force·inv_mass)·dt、inertia = pos + vel_new·dt，得到的是朴素惯性位置 $\mathbf y = \mathbf x + (\mathbf v + \mathbf g\,\Delta t)\,\Delta t$；(2) 它不做 TinyVBD 那套 accelerationComponent 自适应初始化——每个激活顶点都无差别落到 $\mathbf y$；(3) 这是「简洁 vs 更好的热启动」的务实取舍，在叠放/接触场景里会损失一点初始猜测质量，但不影响最终收敛。

惯性位置就位、初始猜测定了。下一节 8.4 进入全篇的核心：把 per-vertex 的 Newton step 在 Warp 里展开，看 solve_elasticity 怎么逐顶点装配 $\mathbf f_i$ 与 $\mathbf H_i$、做 $\Delta\mathbf x = \mathbf H^{-1}\mathbf f$——并顺手点出「没有 Chebyshev」那处差异。
