graph coloring 与并行 sweep

前面四节，我们把 TinyVBD 那根 strand 的每一块零件——惯性初始化、per-vertex 牛顿步、f_i 与 H_i 的装配——都搬进了 Warp。但还差最后、也是最关键的一块：在 GPU 上，「一个顶点一个顶点地扫」根本不能直接照搬。一块 cloth 有几万顶点，串行扫一遍就废了。这一节讲 Newton 怎么用 graph coloring 把这套顺序算法掰成并行，而 min G(x) 一个字没改。

为什么需要着色：races 从哪来

回忆 TinyVBD 的 solve()：它从 iV=1 一路顺序扫到末端，每个顶点就地解一个牛顿步 $\Delta\mathbf x_i=\mathbf H_i^{-1}\mathbf f_i$，立刻写回。这是纯 Gauss-Seidel——处理顶点 $i$ 时，它会用到邻居当前最新的位置。一根 20 顶点的 strand，CPU 上顺序扫一遍快得很，没人在乎。

到了 cloth，问题立刻变味。你想让几万个顶点的牛顿步并行跑——但相邻的两个顶点不能同时更新：它们共享同一个 force element（同一条边、同一个三角），$\mathbf f_i$ 和 $\mathbf H_i$ 的装配都要读对方的位置。两个 thread 一边读一边写同一份数据，就是经典的 data race，结果不可重现、能量也不再单调下降。

VBD 的观察很朴素：只有相邻的顶点才会互相影响。如果两个顶点之间没有任何 force element 直接连着，那它们的 $\mathbf f_i$、$\mathbf H_i$ 谁也不读谁，同时更新毫无冲突。于是只要把「互不相邻」的顶点凑成一批，这一批就能安全并行。怎么找出这些批次？给顶点图染色。

builder.color() → particle_color_groups

在 Newton 里，这件事由建场景时的一行 builder.color() 完成：它给所有 particle（以及 rigid body）做 graph coloring——给每个顶点染一种颜色，要求任何一条 force element 的两个端点颜色不同。染色结果冻进 model.particle_color_groups：一个按颜色分好组的顶点索引列表。同色组里的顶点，两两之间必无 force element 相连。

setup.py
builder = newton.ModelBuilder()
# ... add cloth / particles / shapes ...
builder.color()          # 必需：给 VBD 顶点着色，填 particle_color_groups
model = builder.finalize()
solver = newton.solvers.SolverVBD(model)   # 构造时会检查 particle_color_groups 是否存在

这一步不是可选的优化：SolverVBD 在构造时就会检查 particle_color_groups 是否存在，缺了直接报错。换句话说，对 particle 路径，着色是 VBD 在 GPU 上能跑的前提，不是锦上添花。

Isaac/USD 视角：这一步没有对应物

如果你从 Isaac 过来，会发现 builder.color() 在那边找不到对应概念——它是 VBD 这类逐顶点求解器特有的预处理。Isaac 的 PhysX 走的是另一套求解路径，不需要把顶点按「互不相邻」分批。着色是「primal、按顶点下降」这个内核选择的直接产物。

顶点(primal)着色 vs 约束(dual)着色

这里藏着 VBD 相对 XPBD 的一处结构性便宜。PBD/XPBD 是约束（dual）求解器，它们要并行就得给约束图 / dual graph 着色——保证同色的约束不共享顶点。VBD 是顶点（primal）求解器，它给顶点图着色。

这两张图的稠密程度很不一样。顶点图里，一个顶点的「邻居」就是与它直接连边的那几个顶点——度数往往不大。约束图里，一条约束的「邻居」是所有与它共享顶点的其它约束：一个顶点上挂着好几条边、好几个三角，这些约束彼此就全连成了一团。约束更多、彼此纠缠更密，约束图自然比顶点图稠密得多。而图越稠密，graph coloring 需要的颜色就越多。

结论是定性的、但很关键：对同一张网格，顶点着色用的颜色数明显少于约束着色。颜色少意味着什么，下一节就见分晓——它直接决定一遍 sweep 要分几个顺序阶段。

在一小片 fan 网格上亲手数颜色

「顶点图比约束图稀疏」别只当口号——拿最小的一片 cloth 亲手数一遍，它就变成一个你能复算的几何事实。取一个 fan（扇形）小网格：一个中心顶点 $c$，外圈三个顶点 $v_1,v_2,v_3$，连成 3 个共享中心的三角面（$c v_1 v_2$、$c v_2 v_3$、$c v_3 v_1$）。中心顶点 $c$ 的度数是 3——它跟 $v_1,v_2,v_3$ 都连了边。

先数顶点图（primal）。规则：相邻顶点（有边直连）必须不同色。$c$ 跟外圈三个都相邻，所以 $c$ 要一种颜色（比如 颜色 A）。外圈 $v_1,v_2,v_3$ 沿外环也两两相邻（$v_1\!-\!v_2\!-\!v_3\!-\!v_1$ 是个三角环），三个互相冲突，需要两种颜色错开：$v_1=$B、$v_2=$C、$v_3=$B。一共 3 种颜色（A/B/C）就够——而且无论 fan 再加多少个外圈三角，只要外环是偶数边就能压到 3 色附近，顶点图始终稀疏。

再数约束图（dual）。这里「节点」是 3 个三角面（约束），规则：共享任一顶点的两条约束必须不同色。关键观察——这 3 个三角全都挂在中心顶点 $c$ 上，所以它们两两共享 $c$、两两冲突，在约束图里连成一个完全图（clique）。一个 $N$ 节点的 clique 给每个节点都要一种不同颜色 → 这 3 个三角需要 3 种颜色；推广开：挂在同一中心顶点上的 $N$ 个约束，约束图里就是 $N$-clique，需要的颜色数 = 约束数本身 $N$。

对照就出来了：同一片 fan，顶点图大约 3 色封顶、不随中心扇叶增多而涨；约束图却随挂在中心上的约束数线性涨到 $N$ 色。这就是「顶点着色省颜色」背后可亲手验证的几何根源——中心顶点把所有约束焊成一团 clique，而顶点之间的连接稀疏得多。（具体省多少倍依网格而定，这里只数清机理，不下普适倍数。）

按 color 的并行 sweep

有了颜色分组，Newton 的一遍 sweep 就长成这样（在 _solve_particle_iteration 里）：外层按颜色顺序走，内层一种颜色一次 launch 并行做完。

solver_vbd.py · _solve_particle_iteration
for color in range(len(model.particle_color_groups)):
    # 1) 清零 / 累加该 color 顶点要用到的力与 Hessian
    #    （body-particle contact、spring、self-contact ...）
    # 2) 对这一个 color 的顶点并行解 elasticity 牛顿步
    wp.launch(solve_elasticity, dim=len(color_group), inputs=[...])
    # 3) penetration-free 截断，保证该色 sweep 后不穿透

读这段代码，两层并行/串行的语义要分清：

  • 同一种颜色内部 = Jacobi 风格并行。这一批顶点同时更新，各自读的都是「这一遍开始时」的邻居位置——但因为同色顶点两两不相邻，谁也不读谁刚写的值，所以「用旧值」在这里不损失任何东西，纯赚并行。
  • 颜色与颜色之间 = Gauss-Seidel 顺序。颜色按顺序处理，处理颜色 2 时，颜色 1 的顶点已经写完了新值——后一种颜色读得到前一种颜色刚更新的结果。这正是 Gauss-Seidel 的「立刻代回」，只不过粒度从「单个顶点」放大成了「一整批同色顶点」。

所以颜色数 = 一遍 sweep 里的顺序阶段数 = GPU 同步次数。颜色越少，并行批次越大、同步越少——这就是上面「顶点着色省颜色」为什么是工程基础：它让 cloth 这种大网格的一遍 sweep 只需要少数几次 launch。

同色内用「旧值」为什么不丢精度——这是免费的午餐

看到同色内部是 Jacobi 风格「用这一遍开始时的旧值」，直觉会犯嘀咕：Jacobi 不是比 Gauss-Seidel 收敛慢吗？慢确实存在，但只发生在相邻顶点被迫用对方旧值的时候——少了「立刻代回」那点信息，收敛才打折。可同色顶点按定义就互不相邻：它们之间没有任何 force element，谁的更新都不依赖谁的当前值。于是对它们而言，「用旧值」和「用新值」算出来的 $\Delta\mathbf x_i$ 完全一样——没有任何信息被丢掉。所以这里的并行不是「精度换速度」，而是真正的免费午餐：跨色仍是严格 Gauss-Seidel 拿收敛，同色内并行只是把本就互不依赖的更新凑到一起同时做。

从 TinyVBD 看 把 for iV=1..numVerts 拆成「按 color launch」

TinyVBD 的 solve() 就是一句 for (iV=1; iV<numVerts; ++iV)——一次一个顶点，纯串行 Gauss-Seidel，没有颜色、没有并行批次（参见 3.5，那里我们已经讲过这套着色思想，但也诚实交代了 TinyVBD 自己并不着色）。Newton 做的事，本质上是把这个 for 循环重排：先把顶点按 color 分组，再 for color in color_groups，每个 color 用一次 wp.launch 把这一批顶点一口气并行做完。算法没变——还是逐顶点牛顿步、还是单调下降的 Gauss-Seidel 收敛——只是扫描的次序从「一个挨一个」变成了「一批挨一批」，而每一批内部交给 GPU 的几万个 thread 同时跑。

顺带说一句：这里没有 Chebyshev

particle 路径只有按 color 的 Gauss-Seidel

别去 Newton 的 particle solve 里找 Chebyshev 加速——没有。整套迭代就是「按 color 的 Gauss-Seidel sweep」，重复 self.iterations 次，仅此而已。这和 TinyVBD 不一样：TinyVBD 实现了 Chebyshev（getAcceleratorOmega / applyAccelerator，默认 useAcceleration=false 关着）。所以「TinyVBD 有、Newton particle 没有」是这一篇要反复点名的差异之一。

这一节你要带走的

三句话：(1) GPU 上相邻顶点同时更新会有 data race，builder.color() 给顶点图着色、把互不相邻的顶点凑成同色组，结果存进 model.particle_color_groups（对 particle 必需）；(2) 一遍 sweep = for color in color_groups，同色内部 Jacobi 并行（一次 launch）、跨色 Gauss-Seidel 顺序，颜色数 = 顺序阶段数 = 同步次数，而顶点(primal)着色比约束(dual)着色省颜色，因为顶点图比约束图稀疏；(3) Newton 把 TinyVBD 的串行 for iV 重排成「按 color launch」，min G(x) 与算法本身没变，变的只是扫描次序与并行粒度（且这里没有 Chebyshev）。
