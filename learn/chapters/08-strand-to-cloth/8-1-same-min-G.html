Part 8 · 8-1

同一个 min G(x)，搬上 GPU：什么没变、什么变了

欢迎来到 Newton 篇。这一节是整个 Part 8 的主线，也是你跨过「一根 strand → 一块 cloth」那道坎时唯一需要抓牢的扶手。先把话说死：要解的东西一个字都没变，还是那个 $\min G(\mathbf x)$。变的只是它跑在哪、长什么样、怎么扫。后面五节都在这一句话的延长线上。

那个 G(x)，你早就吃透了

把 1.2 那块地基原样搬过来。一步隐式欧拉，等价于最小化incremental potential：

$$\mathbf x^{t+1} = \arg\min_{\mathbf x}\, G(\mathbf x),\qquad G(\mathbf x) = \sum_i \frac{m_i}{2h^2}\lVert\mathbf x_i-\mathbf y_i\rVert^2 \;+\; E(\mathbf x)$$

左边一项是惯性碗（把每个顶点往预测位置 $\mathbf y_i$ 拉），右边 $E(\mathbf x)$ 是弹性势能。VBD 的解法也没换：一次只动一个顶点 $i$ 的三个自由度，对它的局部能量做一步牛顿，按 Gauss-Seidel 顺序扫过去——

$$\Delta\mathbf x_i = \mathbf H_i^{-1}\mathbf f_i,\qquad \mathbf H_i = \frac{m_i}{h^2}\mathbf I + \sum_{j\in\mathcal F_i}\frac{\partial^2 E_j}{\partial\mathbf x_i^2},\qquad \mathbf f_i = -\frac{m_i}{h^2}(\mathbf x_i-\mathbf y_i) - \sum_{j\in\mathcal F_i}\frac{\partial E_j}{\partial\mathbf x_i}$$

这套公式，就是你在 TinyVBD solve() 里逐行读过的那套，只不过那时 $\mathbf x_i$ 是 2D 平面里的一个点、邻接元件 $\mathcal F_i$ 全是线段弹簧。Newton 的 particle VBD 拿的是同一张谱子，只是换了乐器、换了乐队规模。我们要做的，不是学一个新算法，而是看清这同一个算法在更大的舞台上如何摆开。

interactive回顾：你那根 strand 仍在跑同一个 min G(x)
拖一拖、调调迭代数——下面几节我们把它放大成一块 GPU cloth。

趁手感还热，我们把「放大」这件事拆成四个维度，逐一对照你拖的这根 strand。

放大的四个维度

从 TinyVBD 这根 strand 走到 Newton 的 particle VBD，变化集中在四处——而且每一处都不触碰 $G$ 本身，只触碰「$G$ 长什么样、怎么被扫」。

维度一：2D strand → 3D cloth / soft body

TinyVBD 的顶点活在一个 2D 平面里（重力沿 y 轴），每个 $\mathbf x_i$ 是 2 个自由度，局部牛顿步解一个小系统。Newton 的顶点活在真正的 3D 空间，$\mathbf x_i\in\mathbb R^3$，所以每个 $\mathbf H_i$ 是个 $3\times3$ 矩阵，牛顿步是 $3\times3$ 求逆。仅此而已——惯性项还是 $\frac{m_i}{h^2}\mathbf I$，弹性项还是邻接元件的累加，公式结构纹丝不动。

从 TinyVBD 看 维度多了 1，公式一行没改

你读 solve() 时，惯性项写成 f = m*(inertia-x)/dt²、h = m/dt²*I。Newton 的 particle kernel 里，这两行一字不差，只是 I 变成 $3\times3$、求逆从 colPivHouseholderQr 换成带 $|\det\mathbf H|$ 阈值检查的 $3\times3$ 直接求逆。如果你能在脑子里把那根 strand 从纸面拎起来立到 3D 里，你就已经完成了维度一。

2D→3D 不只是把 H 从 2×2 写成 3×3：为什么 3D 才需要 $|\det\mathbf H|$ 阈值检查

上面那句「求逆带 $|\det\mathbf H|$ 阈值检查」不是随手加的保险，而是维度升上去后真有必要。原因是 $3\times3$ 的局部 Hessian 比 2D（或 strand 的标量/小系统）更容易奇异：一个顶点在 3D 里有三个方向，弹簧 Hessian 在压缩或退化构型下，可能沿多个方向同时塌成近零曲率（比如顶点被压到与邻居共线、共面，三角面退化成一条线）——这时 $\mathbf H_i$ 接近不可逆，$\det\mathbf H_i\to0$，直接 $\mathbf H_i^{-1}\mathbf f_i$ 会吐出爆炸的或乱指方向的 $\Delta\mathbf x$。所以 Newton 的 kernel 在求逆前先看 $|\det\mathbf H_i|$：低于阈值就判定这一步对该顶点不可靠、跳过它这一轮的更新（留给后续迭代和邻居更新把它救回良态）。惯性项 $\frac{m_i}{h^2}\mathbf I$ 本身正定、通常把 $\mathbf H_i$ 兜住，所以这只是退化构型下的兜底；但 2D 里三个方向同时塌的概率低，TinyVBD 用 colPivHouseholderQr 就够，没这一步——这是维度一一句带过的话背后真正的「为什么」。

维度二：弹簧 → membrane / bending / tet 多种元件

TinyVBD 的 $E(\mathbf x)$ 只有一种砖：线段弹簧 $E=\tfrac12 k(l-l_0)^2$（外加一种连 $i$ 与 $i+2$ 的 skip spring，粗略模拟弯曲）。Newton 的 $E(\mathbf x)$ 是一个砖块库：cloth 三角面用 Neo-Hookean membrane、布料折叠用 dihedral bending、soft body 四面体用 volumetric Neo-Hookean，当然也保留了普通 spring。

但请看清这件事的本质：换的只是「$\mathcal F_i$ 里每个元件 $E_j$ 怎么算它那一份 $\mathbf f$ 和 $\mathbf H$」。顶点级的装配逻辑——「把惯性项 + 所有邻接元件的力/Hessian 加起来，再走一步牛顿」——完全没动。多一种元件，就是多一个 @wp.func 往同一个 $\mathbf f_i$、$\mathbf H_i$ 里加料。

从 TinyVBD 看 你的 spring 是这个库里最简单的那块砖

更妙的是，Newton 的 spring 元件，力/Hessian 形式和你 TinyVBD 里写的那个一模一样——同样的 rest length、stiffness，同样的沿弹簧/垂直弹簧两组特征值——就是你在原书 1.5 手算过的那对 $\lambda_\parallel=100$、$\lambda_\perp=-100$（$k=100,l_0=1,l=0.5$），可以点回去拿同一组数复算，确认 Newton spring 和你写的是同一个接口。所以当你看到 Newton 的 membrane 或 bending kernel 时，不必慌：它们和 spring 站在同一个接口后面，都是「给我顶点位置，还我一份 $\mathbf f$ 和一份 $\mathbf H$」。你已经懂了这个接口，剩下的只是元件内部的弹性力学。

维度三：串行 sweep → 按颜色的 GPU 并行 sweep

这是变化最大、也最值得停下来体会的一处。TinyVBD 的 solve() 是一个老老实实的串行 Gauss-Seidel：在 CPU 上从 iV=1 扫到 numVerts-1，一个顶点解完才动下一个，后面的顶点能立刻吃到前面的最新位置。一根 strand 二十个顶点，串行扫一遍眨眼就过。

可一块 cloth 有成千上万个顶点，串行扫会让 GPU 上几千个核心干等着。Newton 的办法是graph coloring：给顶点图染色，同色顶点之间不共享任何 force element，因此可以在 GPU 上同时更新而互不干扰；不同颜色之间则依次进行，保住 Gauss-Seidel 的收敛性质。于是一遍 sweep 变成「逐个颜色：把这一色的所有顶点一起并行解」。

这正是 Warp 的 SPMD 在发力的地方

Newton 建立在 NVIDIA Warp 之上。一个 @wp.kernel 配上 wp.launch(dim=N)，就是一口气开 N 个 thread、每个 thread 用 tid = wp.tid() 认领自己那个顶点。「同色顶点一起解」翻译成 Warp，就是对这一色的顶点数组 launch 一次 kernel。下一节 8.2 会带你看 step() 的全景，8.5 再把着色与并行 sweep 拆开细讲。

从 TinyVBD 看 串行只是「全部一种颜色」的极端情形

别把串行和并行当成两个世界。你那根 strand 之所以能直接串行扫，是因为顶点少、CPU 一个个过最省事。按颜色并行不过是：把「能同时动而不打架」的顶点凑成一组一起动。如果硬要给 strand 染色，相邻顶点共享弹簧、必须错开颜色，扫的次序变了，但每个顶点解的那一步牛顿还是 $\Delta\mathbf x_i=\mathbf H_i^{-1}\mathbf f_i$，一点没变。GPU 不是改了算法，只是换了一个不让核心闲着的扫描次序。

维度四：两处诚实的差异

前三处是「同一个东西放大」。但有两处，Newton 的 particle 路径和你熟的 TinyVBD 确实不一样，必须当面点破，免得你拿 TinyVBD 的记忆去套 Newton 时对不上：

  • 没有 Chebyshev。 TinyVBD 里有 getAcceleratorOmega / applyAccelerator 那套 Chebyshev semi-iterative 加速（回顾 Part 4 讲过）。Newton 的 particle solve 不带它，整套就是按颜色的纯 Gauss-Seidel，没有 overshoot 那一步。
  • 朴素惯性初始化，不是自适应初始化。 TinyVBD 的 forwardStep() 会用 accelerationComponent 做自适应初始化（被托住的顶点别乱猜重力）。Newton 的 particle 路径用最朴素的惯性位置 $\mathbf y = \mathbf x + (\mathbf v + \mathbf g\,h)\,h$ 直接当初值。8.3 会把这两种初始化并排摆出来看。

记住：这两处差异都不动 $G$。Chebyshev 是「怎么更快下到谷底」的加速器，初始化是「从哪儿起步往谷底走」的起点——目标函数 $G(\mathbf x)$ 自始至终是同一个。所以即便去掉 Chebyshev、换个起点，解出来的仍是那同一个 $\min G$ 的最小值点。

这一节你要带走的

一句话，外加一个对照表：要解的还是 $\min G(\mathbf x)=\sum_i\frac{m_i}{2h^2}\lVert\mathbf x_i-\mathbf y_i\rVert^2+E(\mathbf x)$，一个字没变；变的是四件外围的事——维度（2D→3D，$\mathbf H_i$ 成 $3\times3$）、元件（弹簧→membrane/bending/tet，同一个装配接口）、扫描次序（串行 Gauss-Seidel→按颜色 GPU 并行）、以及两处要诚实记住的差异（particle 路径无 Chebyshev、用朴素惯性初始化）。把这根扶手攥紧，下面四节你就只是在看「同一个老朋友换了身行头」。
